1. Жизненный цикл (Lifecycle)
Аналогия: Актёр в театре.

Ваша MainActivity или Fragment — это актёр. У него есть строгий распорядок дня (состояния жизненного цикла). Android-система — это режиссёр, который командует актёром.

onCreate() — элемент подготавливается, но или экрана не существует
onStart() — Экран уже существует, но пользователь пока не может с ним взаимодействовать.
onResume() —Пользователь видит экран, нажимает кнопки, вводит текст. Это активная фаза.
onPause() — Появилось всплывающее окно или зритель отвлекся. Приложение видно, но оно потеряло фокус.
onStop() — Пользователь свернул приложение (нажал кнопку Home) или открыл другой экран.
Экран больше не виден, но он еще жив в памяти телефона.
onDestroy() — Android полностью уничтожил Activity, чтобы освободить память.

2. Зачем нужна ViewModel?
Аналогия: Менеджер за кулисами (или Суфлёр).

Представьте, что пользователь повернул экран телефона. Знаете, что делает Android-режиссёр? Он жестоко убивает актёра (onDestroy()) и мгновенно нанимает нового на ту же роль (onCreate()).

Если ваш актёр (Activity) помнил, что пользователь ввел в поле email, после поворота экрана новый актёр этого знать не будет. Поле очистится.

Здесь появляется ViewModel. Это менеджер за кулисами. Режиссёр (Android) убивает актёров (Activity), но не трогает менеджера (ViewModel). Менеджер живет всё время, пока открыт этот кусок приложения.

Актёр (UI) тупой. Он умеет только рисовать кнопки и слушать нажатия.
Менеджер (ViewModel) умный. Он ходит в базу данных, делает расчеты и хранит введенный email.

3. Подписки и observe (Наблюдатель)
Аналогия: Ресторан (Повар, Официант и Звонок).

Как Актёру (UI) общаться с Менеджером (ViewModel)? Можно заставить UI постоянно спрашивать: "Ну что, юзер зарегистрировался? А сейчас? А теперь?". Это сожрет батарею и повесит систему.

Вместо этого мы используем паттерн "Наблюдатель" (Observer) с помощью LiveData (или StateFlow).

ViewModel (Повар) — готовит еду (обращается к базе данных).
LiveData (Звонок на кухне) — специальная кнопка, которую нажмет повар, когда блюдо готово.
Activity/Fragment (Официант) — стоит и слушает звонок.
observe (Подписка) — это сам процесс ожидания звонка.

Как это выглядит в коде:

// На кухне (во ViewModel):
// Повар ставит пустой поднос со звонком
private val _registrationSuccess = MutableLiveData<Int>() - хранит 1 значение
val registrationSuccess: LiveData<Int> = _registrationSuccess - та же самая коробка (переменная),
но с заблокированной функцией изменения

fun registerUser() {
    // Готовим... (записываем в БД)
    val id = db.userDao().insertUser()

    // ДЗЫНЬ! Кладем ID на поднос и звоним в звонок.
    _registrationSuccess.postValue(id) <- на это событие и подписываемся. Выдача переменной в фоновом режиме
}

// В зале (в Activity/Fragment в методе onCreate/onViewCreated):
// Официант подписывается на звонок
viewModel.registrationSuccess.observe(this) { newUserId ->
    // Этот блок кода сработает ТОЛЬКО тогда, когда прозвенит звонок.
    // Официант берет ID и несет его на другой экран
    navigateToMainScreen(newUserId)
}
Читаем так: "Когда сработает событие, возьми прилетевшее значение, назови его newUserId,
 И ПЕРЕДАЙ ЕГО ВОТ СЮДА (->) в функцию navigateToMainScreen".
